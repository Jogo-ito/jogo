Fluxograma e Lógica de Backend (Supabase)
A arquitetura não possui um servidor dedicado Node.js/Python, operando através de Client-Side Rendering diretamente com o Banco e o Realtime do Supabase.

4.1 Schema de Banco de Dados Implícito
Baseado no código da v1.5, o Supabase precisa conter:

Tabela salas:

id: Texto (Ex: 'A7X9' - Códigos curtos para fácil ingresso).

status: Texto (esperando, sorteado, revelado).

modo: Texto (Padrão: presencial).

Tabela jogadores:

id: UUID gerado pelo Supabase.

sala_id: Foreign Key apontando para salas.

nome: Texto.

numero: Integer (O número sorteado de 1 a 100).

is_host: Boolean.

4.2 Fluxo Lógico e de Estado (State Machine)
Create / Join: O cliente JavaScript faz um INSERT ou SELECT na tabela salas e um INSERT na tabela jogadores.

Inscrição Realtime: Através de sbClient.channel('ito_' + salaAtual), os clientes escutam mutações (postgres_changes) nas tabelas filtradas pelo seu sala_id.

Algoritmo de Sorteio: Quando o Host aperta "Sortear", o próprio cliente JavaScript (frontend do host) gera um Array de 1 a 100, embaralha (usando crypto.getRandomValues para segurança) e executa UPDATE designando um número único para cada jogador no banco. Por fim, altera o status da sala para sorteado.

Re-render Global: A mudança de status na tabela salas engatilha o evento Realtime, que chama a função carregarEstadoSala() em todos os celulares, escondendo o Lobby e revelando a Carta.
